01 Consumer-visible · needs your call changes goldens

Rank base: 0-based or 1-based?

The rank field is emitted 0-based by naming (rank = i) and 1-based by succession (rank = i+1). Same concept, two conventions, both visible in results consumers read. Pick one.

Ranked candidate (a G-major "next chord") naming today succession today
C major — the tonic resolution01
E minor — deceptive12
A minor — predominant return23

Option A — 1-based everywhere

  • A rank emitted to a consumer is an ordinal — "1st, 2nd, 3rd" — not an array index. 0-based reads as an implementation leak.
  • Flip naming to match succession: touches ~25 golden fields (name_pcs, dataset namings).
  • next_chord (71 rank fields, Wend's surface) stays put.

Option B — 0-based everywhere

  • Matches Python indexing; candidates[rank] just works.
  • Flip succession: touches 71 golden fields — the larger diff.
  • Reframes "rank 1 = best" as "rank 0 = best" on the recommendation surface.

Either way: regenerate the two affected goldens in the fix PR, and file a one-line courtesy notice. No consumer has pinned a base in any notice, so this breaks no standing contract — it's tidiness, not a migration.

Recommend

1-based everywhere — ordinal semantics, and the smaller blast radius (~25 vs 71 golden fields, and it leaves Wend's next_chord surface unchanged).

03 Architectural · own PR real value

RE-6b — untangle the session / import cycle. Now, or after #144?

The one RE-6 item with teeth. analysis/builders.py holds a module-level mutable registry (_DEFAULT_SESSION) with import-time disk I/O, and it creates an io ↔ analysis import cycle papered over by ~24 lazy imports across 10 analysis modules.

A registration in one request bleeds into every later call of a long-lived server — the one consumer where session isolation matters most.

The fix — move SessionCatalog below io/ — kills the module-level mutable state and lets the ~24 lazy imports become normal top-of-file imports. Real architectural value, but it touches ten modules, so it's correctly its own PR.

Recommend

Hold until #144 is reviewed, then I take it as a dedicated PR. It's the riskiest change in the review; I'd rather land it clean against a settled main than stack it on an open cleanup branch.
