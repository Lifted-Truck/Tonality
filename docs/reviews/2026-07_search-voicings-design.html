search_voicings — inventing register, exactly

The register-aware sibling of search_identities, and the last piece of the search/ layer. Where identity search asks which pitch-class sets satisfy constraints, voicing search asks how to voice a given one — enumerating concrete registered chords under spacing, bass, and smoothness constraints.

The cardinal-rule crux. Choosing register from a pitch-class identity is the generative act the whole engine is built around not doing by accident. search_voicings does it on purpose, exhaustively, and bounded — it lives in search/ (generative-side), returns a register-bearing Realization, and is never on the analysis path. This is the underexpressed "registered" corner of the data-model lattice, made first-class.

01The shape of the problem

A pitch-class set can be voiced across unbounded octaves — so the search space is infinite until you bound the register. That bound is the central design input: given an identity + a register window, a voicing is one assignment of each pitch class to a specific octave (with optional doublings) inside that window. The space is then finite and exactly enumerable.

The space for Cmaj9 {C E G B D} in C3–C6: each faint tick is a candidate pitch (a pc × octave in the window). A voicing selects one tick per voice — here a closed rooted voicing, spread 14, no ♭9 over the bass. search_voicings enumerates every such selection that satisfies the constraints.

02What it reuses — the layer already holds most of this

Like search_identities, the new code is small: a bounded enumerator + a voicing field vocabulary. The hard parts already exist and are reused, not reimplemented.

Reused (already shipped)

• Condition (eq/in/gte/lte) — the same predicate, now pointed a third way.
• voice_leading_realized — the smoothness primitive (VL cost vs a previous voicing).
• Realization — the register-bearing output type (this is the VOICING spec level).
• voicing_shapes / analyze_voicing — label a result closed / drop-2 / rootless / shell by recognition.

New (this slice)

• The register enumerator — the bounded octave-assignment generator.
• A voicing field vocabulary — spread, bass, intervals-over-bass, doublings, voice count, smoothness.
• VoicingMatch / VoicingSearchResult — typed, with per-voicing metrics for ranking (rule 7).
• A bounded-space guard — a hard cap + truncated, so exhaustive never means explosive.

04The design forks — your calls

Four decisions define the slice. Each carries a recommendation; none is locked.

FORK A

How is the space bounded?

The register must be finite. Options: (a) explicit MIDI window [lo, hi]; (b) register center + max span; (c) voice count + window.

Recommend (a): an explicit register=[lo, hi] window, required. Simplest, exactly bounds the space, and reads directly off the ROADMAP example ("voicings … under 19 semitones of spread"). Center/contour become derived fields, not the bound.
FORK B

Enumerate the full space, or the named-voicing vocabulary?

Full octave-assignment enumeration is exhaustive but larger; the suggest_voicings vocabulary (closed/drop-2/…) is tiny but not exhaustive.

Recommend full bounded enumeration, with labels attached. Exhaustive-and-exact is what makes it search (not suggestion); reuse voicing_shapes to label each result when it matches a named shape. You get completeness and the vocabulary — suggest_voicings becomes a fast special case.
FORK C

Doublings: forbidden, allowed, or a constraint?

Doublings multiply the space (each pc can appear in several octaves). Default matters for both size and musical realism.

Recommend: no doublings by default; a doublings field opts in. Keeps the base space small and predictable; SATB-style doubling is available when asked, bounded by a max-voices cap.
FORK D

Is it stateful across a progression?

Wend's real loop voices chord after chord, each smooth from the last. That needs a from_voicing reference + a VL ceiling — making the query carry prior state.

Recommend: support from_voicing + max_voice_leading as optional fields (via voice_leading_realized), and expose vl_from on every match so the caller ranks by smoothness. This is the field that turns enumeration into a realizer — Wend's realize_voicing becomes a per-step search_voicings call. contour (hold the melodic top) is a v2 field; register-center filtering ships in v1.

05The worked query

The ROADMAP's marquee example plus Wend's smoothness step — the shape of the call and its result.

# "voice Cmaj9 in C3–C6, spread ≤ 19, no ♭9 over the bass, # and stay within 6 semitones of the previous voicing" search_voicings( identity=[0,4,7,11,2], root=0, constraints={"register":[48,84], "spread":{"lte":19}, "no_interval_over_bass":[1,13], "bass_pc":0, "max_voice_leading":6}, from_voicing=[60,64,67,71], limit=50) # → VoicingSearchResult{ count, truncated, voicings:[ VoicingMatch{ # midi:[…], spread, bass_pc, voices, voicing_type?, vl_from } … ] } # every match is a real Realization — register-bearing, exactly enumerated, # ranked by the caller on spread / vl_from (plural outputs, rule 7).
