Tonality · gap 19 · design brief · 2026-07-07

melodic_tendency — which pitch wants to go where, as a cited number

The melodic sibling of next_chord: given a pitch class in a key (optionally over a chord), return the ranked resolutions — where it tends, how strongly, and why. Wend's brief-3 ask, replacing its three hand-set rules (_snap_stable) with a versioned prior; measured evidence already in hand (leap-approached arrivals 65%→26% under the hand rules this formalizes).

The elegant part: the engine already owns the data. Lerdahl's tonal-attraction formula computed over the repo's pinned KK profiles reproduces textbook pedagogy from empirical numbers — ti→do comes out strongest, fa→mi next, and stable tones barely tend anywhere. No new empirical collection; one derivation, frozen into a new versioned prior.

01The model

Lerdahl's anchoring attraction (Tonal Pitch Space; the standard citable model), with stability s read from a tonal-hierarchy table:

a(p → q)  =  (sq / sp) · 1 / d² strength of p's pull toward q = how much more stable the target is, damped by squared semitone distance d. Unstable tones near stable neighbors feel the strongest pull — the leading tone, by construction.

Run live against kk-1982.1 (the profile already pinned in mts/data/) in C major, diatonic step targets:

And the stability ranking itself — C > G > E > F > A > D > B — is the second half of Wend's need verbatim: it replaces the root > third hand rule with a cited ordering (and it's mode-aware for free: the minor profile reorders around ♭3).

02The shape

melodic_tendency(pc_or_degree, key=(tonic, mode), chord=None) → TendencyResult{ source_pc, resolutions: [ {target_pc, strength, distance, evidence[]} … ] # ranked stability: [ {pc, degree, value} … ], # the ranked landing table (root>third replacement) prior_version: "melodic-tendency.1" } # Decision 7: plural, evidenced, version-cited

Analysis-side (mts/analysis/tendency.py + MCP tool). The engine reports pulls with evidence; the caller owns the snap policy — same seam as margin-as-signal. One locked decision (not a fork): the prior JSON freezes its derived stability tables with a documented derivation note ("from kk-1982.1"), rather than reading the profile live — so a future profile default flip can never silently move tendency scales (the versioned-priors lesson, learned once already).

03The forks — your calls

FORK A

Where do the stability numbers come from?

(1) KK-profile-derived — empirical, already pinned in-repo, mode-aware for free; Lerdahl himself calibrated against Krumhansl's data. (2) Lerdahl basic-space integers — theory-set levels (root 4, triad 3, diatonic 2, chromatic 1); cleaner numbers, coarser distinctions (all non-chord diatonic tones tie). (3) Hand pedagogical table — simplest, but uncited numbers are exactly what this gap exists to replace.

Recommend (1) — the table above is real repo data reproducing the pedagogy; frozen into melodic-tendency.1 with the derivation documented. Option (2) can ship later as an alternate prior version if a consumer wants coarse/theory-set behavior — the versioned-prior pattern makes them coexist.
FORK B

What counts as a resolution target?

(1) Diatonic step targets (scale members within ±2 semitones) — anchoring theory's own definition; a leap isn't a resolution. Chromatic sources get their adjacent scale members (usually both semitone neighbors). (2) All diatonic tones at any distance — 1/d² already decays leaps toward zero; more "plural" but reports theoretically-wrong "resolutions" (do→sol as a weak tendency is a leap, not a pull). (3) All 12 pcs — completeness over sense.

Recommend (1) — it is the theory's own target set, and the honest one: reporting a leap as a weak resolution would claim evidence the model doesn't have. The full stability table in the result already gives callers the complete landscape for any other policy.
FORK C

Does chord context ship in slice 1?

Wend's signature includes chord=None, and its hand rules included chord-tone anchoring. Model: when a chord is given, chord tones get a stability boost factor (Bharucha's anchoring — the local stable set), one documented number in the prior JSON; same formula otherwise. (1) Key-only slice 1 — smallest; chord param reserved. (2) Both now — the boost model is ~15 lines and Wend's seam uses it immediately.

Recommend (2) — the marginal cost is one prior field + one multiply, and shipping the signature complete avoids a second consumer round-trip. The boost value is a prior parameter (tunable by version), not code.
